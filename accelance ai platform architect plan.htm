AI Agent Platform

A production multi-tenant agent platform built on a stripped Flowise OSS fork — with NestJS for business logic, Next.js for the product surface, Temporal for durable step execution, and a cloud-agnostic infrastructure stack.

Apache 2.0 Foundation Flowise OSS Fork NestJS API Next.js Temporal.io PostgreSQL + pgvector API Gateway No Upstream Pull
Section 01

Application Architecture

One PNPM monorepo, five independently deployed services. Flowise becomes the AI execution engine — all product logic, auth, and tenancy lives in the NestJS API layer on top. Services communicate over HTTP only; they never import each other's code.

Section 03

Version Roadmap

Each version is a working, deployable release — not a milestone toward something else. v0.0 is the first time customers can use the platform. All subsequent versions add capability without breaking what shipped before. Estimates assume 3 developers working together with normal coordination overhead factored in.

v0.0
First Release — App Ready to Use
Platform Shell
3 weeks
  • Fork Flowise OSS, strip enterprise folder, disconnect upstream permanently
  • Add tenantId to all Flowise entities (~8 tables) — all queries filter by it
  • NestJS auth layer: JWT login, user registration, password reset, email verification
  • Basic RBAC: Admin and Member roles per tenant
  • NestJS proxies all Flowise requests, injecting X-Tenant-Id header
  • Next.js product shell: login page, dashboard, flow builder access, settings
  • Nginx gateway: SSL, routing /* → web, /api/* → NestJS
  • PostgreSQL (with pgvector extension), Redis — production-grade from day one
What customers can do Sign up, log in, build AI flows and agents visually using 200+ integrations, run them, call them via API. Their flows, credentials, and execution history are isolated from all other customers.
v0.1
LLM Management
2 weeks
  • Centralized LLM router module in NestJS — all model calls pass through it
  • Model provider configuration UI (OpenAI, Anthropic, Azure OpenAI, Ollama, custom)
  • Per-tenant model assignment: assign different models to different tenants
  • Fallback chains: if primary model fails or is rate-limited, secondary activates automatically
  • Token budget enforcement with configurable hard cap per tenant tier
  • Token usage dashboard: consumption per tenant, per model, per day
  • Basic guardrails in router: PII scrubbing (regex), rate limiting, token budget check
What customers can do Choose from configured model providers. You control which models are available and at what cost. Token consumption is visible and capped.
v0.2
Workspaces & Teams
2 weeks
  • Workspace entity: one tenant → N workspaces, each fully isolated
  • All Flowise resources (flows, credentials, documents) scoped by workspace_id
  • Team invitations: email invite link → accept → joined workspace
  • Per-workspace RBAC: Owner, Admin, Developer, Viewer
  • Workspace switcher in product UI
  • Personal workspace auto-created on signup
What customers can do Create separate project environments, invite colleagues with specific roles. An Admin in one workspace has no access to another unless explicitly invited.
v0.3
Operational Features
2 weeks
  • Immutable audit log: append-only PostgreSQL table, every action recorded (who, what, when)
  • Audit log never stores raw prompts — PII scrubbed before entry
  • Right to erasure API: cascade delete across PostgreSQL, vector store, S3, conversation history
  • Data retention policies: configurable auto-delete after N days per tenant
  • Agent versioning: save flow versions, restore any prior version in one click
  • Run notifications: failure alert + success summary via email or Teams webhook
  • Usage dashboard: flows run per day, error rates, cost per tenant
What customers can do Have a compliance-grade audit trail. Request full data deletion. Get notified on failures without watching the dashboard.
v0.4
Observability & Guardrails
2 weeks
  • OpenTelemetry SDK instrumented across all five services (traces + metrics + logs)
  • OTel Collector routes to: Prometheus → Grafana, Loki (logs), Tempo (traces), DataDog exporter
  • Langfuse integration: LLM-specific traces — token cost, latency per model, prompt/response pairs
  • Full guardrails pipeline in LLM router: prompt injection detection, content filter on output, topic filter
  • Guardrail trigger rate metrics visible in Grafana
  • Advanced guardrail options: NeMo Guardrails, Llama Guard classifier (per-tenant enable/disable)
What customers can do Connect their existing monitoring stack (Grafana, DataDog, New Relic). See LLM costs and latency per model. Configure guardrails per workspace.
v0.5
Step Execution Engine
4 weeks
  • Temporal cluster deployed (server + UI + PostgreSQL backend)
  • apps/worker service: Temporal worker running step activity implementations
  • Sequential workflow: steps run in order, each checkpointed — failure at step N resumes from N, never re-runs 1 to N-1
  • Per-step retry policy: transient errors (timeout, rate limit) retry automatically; fatal errors escalate immediately
  • HITL approval gates: workflow suspends, approver receives diff of intended write, signal sent on approval
  • Business rule engine: versioned validation rules (date format, status filter, threshold logic) run before/after steps
  • Live execution timeline in web UI (WebSocket step-by-step progress)
What customers can do Build complex multi-step business workflows that checkpoint and resume on failure. Add approval gates before any write-back operation. Define versioned validation rules without changing agent code.
v0.6
Connectors & Testing
3 weeks
  • Python code sandbox: Docker exec with --network none, per-execution isolation, JSON I/O, timeout enforced
  • Connector library: Qlik Sense (API), Microsoft Teams (Graph API), Excel, FortiClient VPN, SAP (as required)
  • HashiCorp Vault integration: all connector credentials move to Vault, short-lived scoped tokens per activity call
  • Testing & simulation framework: mock activity responses, sample data fixtures, dry-run mode (reads live, writes to sandbox)
  • Dev → Staging → Production environment promotion workflow
What customers can do Connect to enterprise data systems. Test agent runs safely against mock data before touching production. Promote validated agents through environments.
v1.0
Enterprise & Distribution
2 weeks
  • SDK: packages/sdk published to npm as @your-org/sdk — typed HTTP client, sk_ secret key auth
  • Widget: packages/widget embeddable chat — UMD bundle, pk_ public key, CORS origin validation
  • SSO: Google OAuth, Microsoft Entra ID — per-tenant SSO configuration
  • Pricing module: token metering, plan tiers (free, professional, enterprise), usage-based billing
  • Data residency configuration per tenant (deploy in specified region)
  • Compliance exports: SOC 2 audit evidence, HIPAA access log exports
What customers can do Integrate via SDK from their own backend. Embed a chat widget on their website. Use their existing corporate SSO. Receive detailed invoices tied to actual usage.

Network isolation Two Docker networks. public_net: gateway only — the single container reachable from the internet. internal_net: all other containers — no inbound internet access. Gateway proxies inbound traffic inward; all outbound goes through the egress allowlist. Engine, Temporal, and Vault are never reachable from outside the internal network.

Environment vCPU RAM Storage Notes

Kubernetes is the recommended production runtime from v0.5 onward — Temporal workers benefit from independent horizontal scaling. For v0.0–v0.4, Docker Compose on a single well-specced VM is sufficient and significantly simpler to operate.
